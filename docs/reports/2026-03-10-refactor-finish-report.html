Refactor Finish Report

Dillinger migration is still in design mode, not implementation mode.

The exact work left is not “finish a partial Next.js port.” It is “turn the approved migration design into a real build plan, then implement the replacement app in a sequence that closes parity gaps before cutover.” The current branch contains one tracked migration artifact and no tracked Next.js scaffold.

Prepared on March 10, 2026 for the current master worktree.
Observed State

Compared to origin/master, the only tracked refactor work is docs/plans/2026-01-19-nextjs-migration-design.md. There is no tracked app/, components/, hooks/, stores/, or lib/ tree.

Immediate Blocker

The design claims “100% functional parity” while explicitly deferring features that the legacy app still ships today: Bitbucket, Google Drive, OneDrive, Medium, Zen mode, scroll sync, and Dropbox image upload.

What Exists

The legacy app is fully mapped enough to start execution: Express entrypoints, Angular modules, localStorage contract, export routes, markdown pipeline, and the cloud plugin surfaces are all identifiable.

Evidence

  • docs/plans/2026-01-19-nextjs-migration-design.md is the approved target architecture.
  • app.js still boots the current Express + EJS + Angular app.
  • views/index.ejs still composes the legacy UI shell.
  • public/js/app.js still bootstraps the active Angular modules.
  • public/js/services/documents.service.js still owns the canonical document/localStorage behavior.
  • plugins/core/server.js still owns /factory/* export and HTML-to-Markdown routes.
  • plugins/github/server.js and plugins/dropbox/server.js are in scope for the approved migration.
  • plugins/bitbucket/server.js, plugins/googledrive/server.js, plugins/onedrive/server.js, and plugins/medium/server.js prove the legacy app has more live integrations than the approved design intends to ship in phase one.
  • The untracked CLAUDE.md files are memory/index artifacts, not refactor implementation.

Bottom Line

Do not start by deleting Angular code or by scaffolding random Next.js files. Finish this refactor by first locking the scope mismatch, then porting the legacy system in dependency order: persistence and markdown core first, shell/editor second, export and cloud routes third, cutover last.

Exact Steps To Finish The Refactor

  1. Resolve the parity contradiction before writing migration code. Update the approved design into an execution spec that marks every currently shipped capability as one of: ship in v1, explicitly defer, or remove. The minimum list that must be decided is: core editor, preview, localStorage compatibility, export routes, HTML-to-Markdown, GitHub, Dropbox, Bitbucket, Google Drive, OneDrive, Medium, Zen mode, scroll sync, and Dropbox image upload.

    Exit criterion: there is a single parity matrix with no “implied later” ambiguity.

  2. Choose the real repository topology and freeze it. The design doc shows a sibling-style dillinger-next/ layout, but the current repo has no such scaffold. Decide whether the Next.js app lives in this repo root, under a subdirectory, or in a separate repo/worktree. Do not continue until the directory layout, package manager, Node version, and deployment target are fixed.

    Exit criterion: one committed scaffold decision with a booting Next.js app.

  3. Write the migration backlog from the legacy inventory, not from memory. Convert the current Angular/Express surfaces into implementation tickets. The canonical source files are: views/index.ejs, public/js/app.js, public/js/base/base.controller.js, public/js/services/documents.service.js, public/js/user/user.controller.js, plugins/core/server.js, and each plugin server/service pair.

    Exit criterion: every legacy surface is mapped to a target component, hook, store slice, or API route.

  4. Port the persistence contract before the UI shell. Implement the new store and hydration layer first, because everything else depends on it. Preserve the legacy localStorage keys files, currentDocument, and profileV3. Preserve the default document semantics from public/js/factorys/sheet.factory.js, including the seeded untitled document and README-derived starter content.

    Exit criterion: create/select/delete/save flows work in the new app with legacy storage data loaded in place.

  5. Extract and lock markdown parity next. Rebuild the markdown pipeline from plugins/core/markdown-it.js, including the plugin set, KaTeX, syntax highlighting, table classes, and line-number attributes used by preview/scroll behavior. Generate fixture snapshots from the legacy renderer before swapping in the new one.

    Exit criterion: the new renderer matches the legacy HTML contract for representative markdown fixtures.

  6. Build the editor shell only after store and markdown are stable. Port the shell composed today by views/index.ejs, views/navbar.ejs, views/sidebar.ejs, views/title.ejs, views/editor.ejs, and views/preview.ejs into the target Next.js component tree. Replace Ace with Monaco, but preserve the current UX contract: editor/preview split, document switching, autosave, settings, counts, raw HTML preview mode, and keyboard modes.

    Exit criterion: the React shell can replace the Angular shell for a local user without touching export or cloud features yet.

  7. Port the export and conversion endpoints before cloud integrations. Recreate /factory/fetch_markdown, /factory/fetch_html, /factory/fetch_pdf, and /factory/html_to_md as Next.js route handlers or an equivalent server layer. Decide whether to preserve the legacy route shapes for compatibility or migrate the client fully to /api/export/* and /api/convert/html-to-md.

    Exit criterion: markdown, HTML, styled HTML, PDF, and HTML-to-Markdown all work end to end from the new UI.

  8. Port GitHub and Dropbox as the first cloud integrations. This matches the approved design and captures the most important external workflows. Port both the API routes and the client state/modal flows now living in plugins/github/*, public/js/plugins/github/*, plugins/dropbox/*, and public/js/plugins/dropbox/*. Include GitHub org/repo/branch/tree selection, file fetch, SHA tracking, and save. Include Dropbox auth, search, fetch, save, and decide whether Dropbox image upload is v1 or explicitly deferred.

    Exit criterion: a user can authenticate, open remote markdown, edit it, and save it back in both systems.

  9. Close or formally defer the remaining legacy integrations. Bitbucket, Google Drive, OneDrive, Medium, Zen mode, scroll sync, and Dropbox image upload are live legacy behavior today. If they are not shipping in the first Next.js release, document that as a product decision, not as a silent omission. If they are shipping, port them now before calling the refactor complete.

    Exit criterion: there are no undocumented feature regressions left.

  10. Build a parity verification suite before cutover. The repo currently has meaningful tests only around Dropbox. Add a migration test suite that covers store hydration, markdown fixtures, export routes, settings behavior, and at least mocked GitHub/Dropbox happy paths. For browser behavior, add end-to-end checks for document lifecycle, preview refresh, and export flows.

    Exit criterion: you can prove parity with automated checks instead of hand-testing from memory.

  11. Run side-by-side cutover and remove the legacy app last. Keep the Express/Angular implementation as the reference system until the new app passes the parity matrix. Only then wire production traffic to the new app, migrate any required environment variables, and archive or remove the legacy code paths.

    Exit criterion: the new app is the default runtime and the legacy app is no longer serving user traffic.

Recommended First Commit Sequence

If you want the shortest path to forward motion without rework, the first commits should land in this order:

  1. Update the migration doc with a parity matrix and explicit defer list.
  2. Scaffold the Next.js app and commit the runtime/tooling baseline.
  3. Port the store, hydration, and markdown renderer with fixture tests.
  4. Port the editor shell and document lifecycle.
  5. Port export routes, then GitHub and Dropbox.
  6. Either port or formally de-scope the remaining legacy integrations.
  7. Add parity tests and cut over only after they pass.
